# 项目结构与设计意图说明

## 项目定位

本项目是一个面向计算机网络实验课的 AI 助教系统。它的目标不是做通用问答，而是围绕课程实验文档、实验拓扑和教学引导策略，帮助学生完成网络理论理解、实验故障排查、配置审查和计算题辅导。

系统的核心设计可以概括为：

- 以课程文档为证据来源，通过 RAG 检索减少模型幻觉。
- 以结构化拓扑数据补充实验场景，支持和设备、接口、链路、VLAN、IP 规划相关的问题。
- 以分层提示策略控制回答深度，尽量采用苏格拉底式引导，而不是默认直接给最终答案。
- 以 FastAPI、Streamlit 和 MCP 形成不同接入方式，兼顾本地演示、API 调用和外部 Agent 工具调用。

## 顶层目录

项目根目录主要包含以下部分：

- `agentic_rag/`：核心智能体、RAG、拓扑检索、Prompt、LLM 封装等逻辑。
- `server.py`：FastAPI 后端入口，也提供给 Streamlit 直接调用的本地函数入口。
- `app_streamlit.py`：Streamlit 前端原型，负责登录注册、会话列表、聊天展示、图片上传和反馈交互。
- `storage/`：用户、会话、反馈、交互指标和学生水平评分的持久化逻辑。
- `mcp_server/`：只读 MCP Server，用于向支持 MCP 的客户端暴露课程检索和拓扑查询能力。
- `components/`：Streamlit 自定义前端组件，主要用于支持带图片的聊天输入。
- `scripts/`：启动脚本、服务安装脚本、拓扑数据构建脚本和模型下载脚本。
- `eval/`：评测、消融实验、题库构建、拓扑评测和性能测试脚本。
- `data/`：课程实验指导书 `.docx` 原始资料，是文本知识库的来源。
- `docs/`：系统架构图、论文图、Mermaid/DrawIO 图等设计材料。
- `test_examples/`：测试样例数据。

## 核心智能体层：agentic_rag

`agentic_rag/` 是系统最重要的目录，承载了“检索增强 + 工具调用 + 分层教学”的核心逻辑。

### `agent.py`

`agentic_rag/agent.py` 是主 Agent 流程。

它负责：

- 接收用户问题和历史对话。
- 判断问题是否属于计算机网络课程相关范围。
- 将问题分类为实验故障排查、理论概念、配置审查、计算分析等类别。
- 根据对话状态决定 `hint_level` 是否保持、提升或直接跳到最高级。
- 识别当前实验编号，例如 `实验13` 或 `lab13`。
- 组装最终 system prompt。
- 让模型通过 `<tool_calls>` 协议调用工具。
- 执行 `检索`、`拓扑`、`搜索` 等工具，并把结果注入下一轮推理。
- 收集引用和工具调用轨迹。
- 返回最终回答、更新后的历史和新的状态。

这个文件体现了项目的核心设计：模型不是直接回答，而是在一个受控的工具调用循环中工作。

### `prompts.py`

`agentic_rag/prompts.py` 定义了系统的教学策略。

主要包括：

- 相关性判断 Prompt。
- 问题分类 Prompt。
- hint level 决策 Prompt。
- 四类问题的 base prompt。
- 四类问题的分层教学策略。

四类问题分别是：

- `LAB_TROUBLESHOOTING`：实验故障排查。
- `THEORY_CONCEPT`：理论概念与原理。
- `CONFIG_REVIEW`：配置操作与审查。
- `CALCULATION`：计算与分析。

每类问题都有不同的回答目标和约束。例如故障排查强调分层诊断，理论概念强调定义和类比，配置审查强调拓扑核对和错误定位，计算题强调分步引导。

### `rag.py`

`agentic_rag/rag.py` 实现课程文档 RAG。

主要职责是：

- 从 `data/` 下的 `.docx` 加载实验指导书。
- 将文档切分为 chunk。
- 构建或加载 FAISS 向量索引。
- 使用 HuggingFace embedding 模型做 dense retrieval。
- 使用 BM25 做关键词召回。
- 通过 RRF 融合 BM25 与 dense 结果。
- 可选使用 CrossEncoder reranker 进行重排。
- 根据问题类别和 `hint_level` 调整检索参数。
- 返回带引用的检索上下文。

项目的文本检索不是固定参数，而是教学意图感知的检索。例如不同题型、不同提示层级会影响召回数量和返回证据数量。

### `topo_rag.py`

`agentic_rag/topo_rag.py` 负责拓扑结构化数据的构建和运行时读取。

拓扑数据的设计分为两个阶段：

- 离线构建阶段：从 `.docx` 中抽取图片，判断是否为拓扑图，调用视觉模型抽取设备、接口、链路、IP、VLAN、子网等结构化信息，再经过审核后落盘。
- 在线运行阶段：只读取 `approved_json` 下审核通过的 JSON，不在聊天时临时解析图片。

这种设计的意图是降低在线回答的不确定性。视觉模型可能会误读拓扑图，因此项目通过“抽取、审核、发布”的流程，把不可靠数据挡在运行时之外。

### `topo_models.py`

`agentic_rag/topo_models.py` 定义拓扑结构化数据的 Pydantic schema。

核心数据结构包括：

- `Device`：设备。
- `Interface`：接口。
- `Link`：链路。
- `Subnet`：子网。
- `TopologyExtraction`：完整拓扑抽取结果。
- `TopologyDraftArtifact`：初次抽取产物。
- `TopologyReviewArtifact`：审核产物。
- `TopologyBuildManifest`：构建过程清单。

这些 schema 保证拓扑数据结构稳定，便于后续检索、格式化和 MCP 暴露。

### `llm_config.py`

`agentic_rag/llm_config.py` 封装聊天模型调用。

当前默认使用 DeepSeek 风格的 `/chat/completions` 接口，支持普通调用和流式调用。它把 LangChain message 对象转换为 OpenAI 兼容格式，并通过环境变量读取 API key、base URL 和模型名。

### `web_search.py`

`agentic_rag/web_search.py` 是可选联网搜索工具。

它延迟初始化 DuckDuckGo 搜索工具，只在 Agent 决定调用 `搜索` 时执行。Prompt 中明确要求搜索只能作为课程文档和拓扑无法覆盖时的补充，不能优先用于回答实验配置问题。

### `vision.py`

`agentic_rag/vision.py` 用于图片文字识别。

虽然文件名叫 `vision`，但当前实现主要是 OCR：使用 `rapidocr_onnxruntime` 从用户上传图片中提取文字，再把识别结果拼进用户问题交给 Agent。

### `chat_format.py`

`agentic_rag/chat_format.py` 用于拆分模型回答中的隐藏思考和可见内容。

它识别 `<思考>...</思考>` 和 `<thinking>...</thinking>` 标签，将内部推理与最终面向学生的回答分离。Streamlit 和 API 都会用它避免把内部推理直接暴露给普通回答内容。

## 后端服务：server.py

`server.py` 是 FastAPI 后端，也是 Streamlit 的本地调用入口。

它提供：

- 用户注册和登录。
- Bearer token 鉴权。
- 普通聊天接口 `/api/chat`。
- SSE 流式聊天接口 `/api/chat/stream`。
- 会话查询、删除、归档等接口。
- 点赞、点踩和取消反馈。
- 健康检查接口。

同时，它也暴露了给 Streamlit 直接调用的函数，例如：

- `chat_once`
- `chat_once_stream`
- `register_user`
- `login_user`
- `submit_feedback`
- `load_user_session_cache`

这说明项目有两种使用方式：

- 作为 HTTP API 服务运行。
- 作为本地 Streamlit 应用直接 import 后端函数运行。

`server.py` 还负责会话锁、并发限制、历史压缩、摘要更新、异步记录指标和学生水平更新。

## 前端原型：app_streamlit.py

`app_streamlit.py` 是 Streamlit 前端。

它实现：

- 登录和注册 UI。
- 会话列表。
- 新建、删除、归档和恢复会话。
- 聊天消息展示。
- 流式输出展示。
- 隐藏思考内容的折叠显示。
- Debug 模式下的工具调用详情展示。
- 点赞、点踩反馈。
- 图片上传和粘贴。
- OCR 后把图片文本拼接进用户问题。

它不通过 HTTP 调用 `server.py`，而是直接 import 后端函数。这种设计适合本地演示和课程原型部署，减少前后端联调复杂度。

## 存储层：storage

`storage/` 管理用户、会话、反馈、交互日志和学生水平。

### `auth.py`

`storage/auth.py` 提供基础认证工具。

它负责：

- 用户名格式校验。
- 密码格式校验。
- PBKDF2 密码哈希。
- 密码校验。
- token 生成。
- token 过期时间计算。

### `user_store.py`

`storage/user_store.py` 是主要持久化模块。

它使用 SQLite，数据库默认位于 `data_store/app.db`。

主要表包括：

- `users`：用户信息。
- `tokens`：登录 token。
- `sessions`：会话、摘要、历史、状态。
- `logs`：用户行为日志。
- `message_feedback`：消息点赞点踩。
- `interaction_metrics`：单轮交互指标。
- `proficiency_scores`：学生水平评分。

它还包含从旧 JSON 文件迁移到 SQLite 的逻辑。

### `proficiency.py`

`storage/proficiency.py` 是学生水平评分引擎。

它不调用 LLM，而是根据交互状态做纯规则计算：

- 如果 hint level 上升，说明学生可能需要更多帮助，水平信号偏负。
- 如果低 hint level 能保持，说明学生可能能跟上引导，水平信号偏正。
- 使用 EMA 和时间衰减更新分类分数。
- 根据分类分数和总体分数生成学生水平摘要。
- 新会话可根据学生水平决定初始 `hint_level`。

这个模块把“教学个性化”从抽象口号落到了可持久化、可计算的状态更新上。

## MCP 服务：mcp_server

`mcp_server/` 是只读 MCP Server。

它暴露的工具包括：

- `retrieve_course_docs`：检索课程文档。
- `get_topology_context`：读取指定实验拓扑上下文。
- `list_available_experiments`：列出已有拓扑数据的实验。
- `get_experiment_manifest`：读取实验 manifest 或基于 approved JSON 返回摘要。

MCP 层调用的是现有 `rag.py` 和 `topo_rag.py` 能力，不改变主 API 和前端链路。它的设计意图是让 Cursor、Claude Desktop 或其他支持 MCP 的客户端能复用项目中的稳定检索能力。

## 自定义组件：components

`components/` 中目前主要有聊天输入组件。

`components/chat_input_images/` 提供一个 Streamlit 自定义组件，用于实现类似 ChatGPT 的输入栏，支持：

- 文本输入。
- 图片粘贴。
- 图片上传。
- 缩略图展示。

组件返回 JSON 字符串，包含用户文本和图片 base64。Streamlit 收到后调用 OCR，再把图片文字与用户问题合并。

## 脚本：scripts

`scripts/` 包含启动和构建相关脚本。

常见脚本包括：

- `start_dev.sh`：开发模式启动后端和 Streamlit，启用后端快速启动和后台 RAG 预热。
- `start_all.sh`：稳定模式启动，等待后端和 RAG 就绪后再启动前端。
- `start_mcp_server.sh`：启动 MCP Server。
- `build_topology_store.py`：批量或单个构建拓扑结构化数据。
- `download_bge_m3.py`：下载 embedding 模型。
- `install_service.sh` 和 `rag-agent.service`：服务化部署相关。

## 数据与索引

`data/` 存放课程实验指导书 `.docx`。

RAG 默认从这些文档构建或加载索引。项目 README 中还提到：

- `faiss_index/`：向量索引目录。
- `topo_store/`：拓扑结构化产物目录。
- `data_store/`：SQLite 数据目录。

这些目录不一定都在当前文件列表中出现，因为部分内容可能是运行后生成或被 `.gitignore` 忽略。

## 评测目录：eval

`eval/` 用于论文或实验验证。

它包含：

- 消融实验。
- 性能基准测试。
- 苏格拉底式引导评测。
- 学生水平评测。
- 拓扑问答评测。
- 题库构建和清洗脚本。
- 检索实验对比脚本。

这说明项目不是只做一个 demo，而是围绕检索质量、拓扑增强、教学引导效果和系统性能做了评估设计。

## 主要运行流程

一次典型聊天请求大致如下：

1. 前端或 API 把用户问题发给 `server.py`。
2. `server.py` 加载会话历史、摘要和状态。
3. `server.py` 调用 `agentic_rag.agent.query` 或 `query_stream`。
4. `agent.py` 做相关性判断、问题分类和 hint level 决策。
5. `agent.py` 根据分类选择对应 prompt 和策略。
6. 模型如果需要证据，会输出 `<tool_calls>`。
7. 系统执行课程文档检索、拓扑读取或联网搜索。
8. 工具结果作为下一轮输入交给模型。
9. 模型生成最终回答。
10. 系统拆分隐藏思考和可见内容。
11. 系统保存会话、工具轨迹、交互指标和学生水平变化。
12. 前端显示可见回答，Debug 模式下可查看工具调用详情。

## 设计意图总结

这个项目的架构重点不是“把 LLM 接进聊天框”，而是构建一个可控的教学 Agent。

它通过 RAG 控制事实来源，通过拓扑 JSON 控制实验上下文，通过 prompt 策略控制教学风格，通过 `hint_level` 控制帮助强度，通过 SQLite 保存长期状态，通过 MCP 暴露只读能力。

整体设计体现了几个明确取向：

- 证据优先：回答应尽量来自课程文档或审核通过的拓扑数据。
- 教学优先：根据学生状态分层引导，而不是一上来给完整答案。
- 场景优先：重点服务计算机网络实验课，而不是泛化到所有知识问答。
- 可追踪：保留引用、工具轨迹、会话历史和交互指标。
- 可评估：保留评测脚本和题库，用于验证检索、拓扑增强和教学策略效果。
- 可扩展：FastAPI、Streamlit 和 MCP 分别服务 API、前端原型和外部 Agent 接入。
